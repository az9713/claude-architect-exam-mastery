MCP Server Scoping Hierarchy (TS 2.4)
Scoping Rules: Project-scoped (.mcp.json) = shared team tooling, committed to version control. User-scoped (~/.claude.json) = personal/experimental servers, not shared. All configured MCP servers are discovered at connection time and available simultaneously.
// .mcp.json — Project-scoped, version-controlled
{
  "mcpServers": {
    "github": { "type": "stdio", "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } },
    "jira": { "type": "http", "url": "${JIRA_URL}", "env": { "API_KEY": "${JIRA_API_KEY}" } }
  }
}

// ~/.claude.json — User-scoped, personal/experimental
{
  "mcpServers": {
    "my-local-db": { "type": "stdio", "command": "local-db-server" }
  }
}
MCP Tools vs MCP Resources (TS 2.4)

MCP Tools

  • Execute actions with side effects (query, create, update)
  • Called on-demand by the agent when needed
  • Return results after execution
  • Examples: search_issues, create_ticket, query_database
  • Each call is an explicit agent action

MCP Resources

  • Expose content catalogs — read-only structured data
  • Give agent visibility into available data upfront
  • Reduce exploratory tool calls needed
  • Examples: issue summaries, doc hierarchies, DB schemas
  • Agent can browse what's available before querying
When to use Resources: When you want to give the agent a catalog of available content without requiring tool calls to discover it. For example, exposing a list of open Jira issues as a resource means the agent can browse them without calling search_issues iteratively.
