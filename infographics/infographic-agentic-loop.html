Agentic Loop & Multi-Agent Architecture

Domain 1 — Task Statements 1.1, 1.2, 1.3

The Agentic Loop Lifecycle (TS 1.1)
Key Rule: The loop ONLY terminates on stop_reason = "end_turn". Never parse natural language to detect completion. Never use arbitrary iteration caps as the primary stopping mechanism.
Anti-Pattern: Natural Language Parsing

Checking if assistant text contains "I have completed the task" — the model may phrase completions differently or include these phrases mid-task.

Anti-Pattern: Arbitrary Caps

Terminating after N iterations as the primary mechanism — a complex task may legitimately need more iterations than an arbitrary limit.

Anti-Pattern: Content Checking

Checking whether assistant response contains tool_use blocks to determine completion — use the structured stop_reason field instead.

Hub-and-Spoke Multi-Agent Architecture (TS 1.2)
Coordinator (Hub)
Subagent (Spoke)
Delegation (→)
Results (←)
Hub-and-Spoke Rule: ALL inter-subagent communication routes through the coordinator. Subagents operate with isolated context — they don't inherit the coordinator's conversation history and don't communicate directly with each other. The coordinator handles task decomposition, delegation, result aggregation, and error recovery.
Parallel Subagent Spawning (TS 1.3)
Context Passing Rule

Subagents receive NO automatic context. All findings from prior agents must be explicitly included in the subagent's prompt. Pass web results, document summaries, and metadata directly.

AgentDefinition Fields

Each subagent type is configured with: description, systemPrompt, allowedTools, and tool restrictions. The description helps the coordinator decide which subagent to invoke for each task.

Goal-Based Prompting

Coordinator prompts should specify research goals and quality criteria, not step-by-step instructions. This enables subagent adaptability when the expected path hits obstacles.
