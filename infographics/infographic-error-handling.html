Error Category Taxonomy (TS 2.2)
TRANSIENT
Action: RETRY

Temporary failures that may succeed on retry. Service is functioning but temporarily unavailable.

Examples: database timeout, network blip, rate limit (429), service overload (503)
Schema:
isError: true
errorCategory: "transient"
isRetryable: true
retryAfterMs: 2000
VALIDATION
Action: FIX INPUT

Client sent invalid data. Retrying with same input won't help — the input must be corrected.

Examples: invalid email format, missing required field, value out of range, malformed JSON
Schema:
isError: true
errorCategory: "validation"
isRetryable: false
invalidField: "email"
BUSINESS
Action: EXPLAIN

Request violates business rules or policy. Not a technical error — must be communicated to user.

Examples: refund exceeds limit, order outside return window, policy doesn't allow this action
Schema:
isError: true
errorCategory: "business"
isRetryable: false
customerMessage: "..."
PERMISSION
Action: ESCALATE

Action requires elevated permissions or human authorization. Agent cannot self-authorize.

Examples: requires manager approval, above agent authority threshold, compliance escalation needed
Schema:
isError: true
errorCategory: "permission"
isRetryable: false
escalateTo: "manager"
Anti-Pattern: Generic "Operation failed" responses prevent intelligent recovery. The coordinator has no basis to distinguish a retry-worthy timeout from a policy violation. Structured error metadata (errorCategory, isRetryable) enables programmatic recovery decisions.

Error Propagation in Multi-Agent Systems (TS 5.3)
Key Distinction: Access failures (database timeout, service unreachable) need retry decisions. Valid empty results (query succeeded, no matches found) represent successful completion. These must return distinct structured responses — both returning an empty array makes them indistinguishable to the coordinator.
